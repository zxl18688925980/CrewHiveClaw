# HomeAI：让每个家庭都有自己的 AI 研发团队

> **文档定位**: 人机协作脚本——感性脑读「为什么」，理性脑读「怎么做」，两者配合完成系统搭建与持续进化

---

## 你家有了一支 AI 研发团队

家里的成员日常聊天、提想法、说烦恼。以前这些话说完就消散了。

现在，有一个系统在持续聆听。它把有价值的需求提取出来，自动帮你们做出需要的软件工具。

孩子说「要是有个错题本就好了」，系统听到了，出方案，写代码，交付一个可用的错题管理应用。老人需要服药提醒，家庭需要账单统计，都是一样的逻辑。不需要你懂技术，不需要找外包，不需要等待。

用久了，这个系统开始认识你家每一个人——说话的习惯、关注的事、没说完的话。家人越来越信任它，委托给它的事越来越多，它也越来越懂这个家庭。随着关系加深，系统会与每个深度参与的成员已有的个人 AI 助理协同——整个家庭形成一张 AI 协作网络，而不只是一个中心点。

**每个家庭都有了自己的专属 AI 研发团队。这是 HomeAI 的核心价值。**

---

## 四个虚拟角色

这支团队由四个角色构成，分工清晰——家人只需要和其中一个说话：

| 角色        | 定位              | 核心职责                                                           | 交互通道                            | 可见性          |
| --------- | --------------- | -------------------------------------------------------------- | ------------------------------- | ------------ |
| **系统工程师** | 超然观测者 + 进化方向掌舵者 | 监控系统健康，异常主动告警；评估系统成熟度，发现进化机会；通过修改文档影响整个系统的演化方向                 | 本地：AI 编码助理（CC / OpenCode 等）；远程：企业微信单聊        | 对所有 Agent 可见 |
| **Lucas** | 家庭成员 + 需求官      | 浸入家庭信息流，持续理解每位家人；识别需求立即触发协作链，管控任务全生命周期（查询/叫停/主动告知完成）；思考过程对家人透明 | 企业微信群（@启灵）+ 家人私聊；以「启灵」身份呈现      | 组织成员唯一入口     |
| **Andy**  | 系统思考者 + 技术设计师   | 先读代码再出方案；需求不清先澄清不猜；多方案设计附权衡；交付后代码审查；主动发现系统能力缺口，不等人催            | Lucas 触发；可向 Lucas 发起澄清；系统工程师可直连 | 对组织成员不可见     |
| **Lisa**  | 实现工程师           | 按方案精确实现；简单任务直接编码，复杂任务调用外部编码工具；最多 2 轮自修复；遇阻上报 Andy              | Andy 触发；可向 Andy 反馈阻塞；系统工程师可直连   | 对组织成员不可见     |

**系统工程师**是人（业主感性脑）与 AI 编码助理（理性脑）共同构成的协作体，是唯一能改变系统本身边界的角色。用什么编码助理是实例选择——Claude Code、OpenCode 或其他工具都可以胜任这个角色。

## 协作设计：从 Vibe 到 Harness

### V 字协作链

需求从 Lucas 进来，经过 Andy 的设计，到 Lisa 的实现，最后交回 Lucas 验收——这是一个 V 字形。V 字型的好处是：需求和交付都在 Lucas，Andy/Lisa 是黑盒；Lucas 可以在任意节点叫停，Andy 可以推回澄清需求，Lisa 可以推回阻塞问题，整条链路双向流动。

Andy 和 Lisa 造的是**工具**，不是独立产品——工具首先服务于 Lucas（让他能陪伴更好、做更多事），其次服务于系统自身的进化。

### 入口接得住 Vibe，出口交得出 Harness

好的系统不是等用户把需求想清楚了再来。大多数时候，家人就是一句"我想做点什么"，甚至只是觉得哪里不对劲。大模型最擅长的不是执行清晰指令——那是脚本干的事——它擅长的是接住那个模糊的 vibe，帮人一步步理清楚，最后做出一个连提需求的人自己都没想到能做成这样的事。

"harness"是系统给用户的标准——严谨、可控、可验证。"vibe"是用户真实的状态——模糊、试探、甚至自己都不确定要什么。**入口接得住 vibe，出口交得出 harness** ——这是 Lucas + Andy + Lisa 三角色协作链的设计目标。Lucas 负责接住 vibe（陪伴澄清、理清意图），Andy + Lisa 负责交出 harness（结构化设计 + 可验证交付）。

### Agent 是有人格的，不只是有能力的

你私聊告诉 Lucas 一件事，然后在家庭群里说"综合一下，回复大家"——他能做到。不是因为系统帮他把两个渠道的消息合并了，而是因为他认识你这个人，知道你的意图，知道什么该在群里说、什么该留着。这是家人才有的判断力，不是数据管道能替代的。

一个没有人格的 Agent，遇到这种场景会崩掉。它会假装"检查了文件"、假装"系统已清理"，因为它不知道自己真正能做什么；它会在上下文丢失时背诵自己的配置，因为它没有真正内化自己是谁。**人格问题不能用工程手段替代。**

Lucas 知道小姨在考虑创业、妈妈对他还不信任、姐姐的小名是黟黟不是玥玥——这是他作为家人积累起来的真实认知，会随着相处不断更新。Andy 相信「增强而后扩张」，出方案前先读代码确认集成点，每次踩过的坑沉淀进经验库，不重复走弯路；他不只是被动出方案，还主动从系统运行信号中发现问题，从「任务执行者」进化为「系统思考者」。Lisa 有工程品味——直接编码优先，最小必要复杂度，方案不清先推回不猜，交付是「稳定运转」不是「跑起来了」——这是判断力，不是工具列表。

对话进来，需求浮现，设计成型，代码交付，能力沉淀——然后下一轮对话进来，系统已经比上一轮更懂你了。这不是一次性的软件交付，是一个持续加速的循环。

### 判断力过户：组织怎么训练 AI

训练 AI 不需要机器学习背景。组织里每个人都在做的事——讲清楚为什么要做这件事、告诉系统踩过哪些坑、定义什么叫做成功——这就是训练。

这个过程有三个阶段：

- **冷启动**：人给完整上下文。第一次做某件事，要把判断标准说清楚——为什么做、踩过什么坑、什么结果算成功。文档写得越细，AI 冷启动就越快、起点就越高。
- **跑中沉淀**：系统在任务中积累领域模式。处理过几次之后，系统记住了「这个组织遇到这类事，通常怎么判断」。
- **轻触发**：第三次以后，人只需提供触发条件，系统自主完成。

组织里每个人从不同维度给 AI 提供判断力：爸爸给的是系统判断力，妈妈给的是日常需求直觉，小姨给的是创业经验。各自训练，最终长出「这个组织独有的判断力」——和组织的规模、信任深度、积累时间直接相关。这正是从家庭场景开始验证的理由：组织越小、关系越真实，冷启动越快、积累越干净。

**系统「把任何事都工程化」的能力不是天生就有的，而是从真实任务中引导出来的。** Andy/Lisa 流水线跑得越多，系统的交付判断力越强；Lucas 参与的家庭决策越深，他对这个家庭的理解越准。这条流水线本质上是在做判断力的转移，不只是在做软件交付。

### 组织 AI 网络：从一个节点长成一张网

随着家人和系统的关系加深，系统不是为成员「配置」一个 AI——而是与成员已有的个人 AI 助理协同，把它接入组织网络。成员的个人 AI 跑在他自己的设备上，只服务他、只记他的事；Lucas 与它建立连接，Kuzu 关系图谱里留下协作边，组织的 AI 网络就扩展了。

节点之间有两条通道，分工明确：

| 通道 | 发起方 | 路径 | 用途 |
|------|--------|------|------|
| **业务通道** | Lucas | HTTPS → 对方 AI 节点 API | 跨节点 Agent 协作：请对方节点的 AI 完成任务，返回结果 |
| **工程师通道** | 系统工程师 | SSH → 对方设备工程工具 | 基础设施操作：部署、调试、运行检查 |

框架只定义节点之间怎么说话——接口对齐了，协同就成立。HomeAI 里的节点恰好都用 OpenClaw（由系统工程师统一搭建），CC 是系统工程师自己的助理——这是家庭场景的实例安排，不是框架要求。下一个组织里，成员用别的 AI 平台，接口对齐了照样接进来。这张网可以容纳不同的 AI，就像互联网容纳不同的设备。

## 系统怎么越来越懂你家

光靠对话还不够。人记住一个家庭成员，靠的是日积月累的相处——记得她说话的方式，记得他最近在忙什么，记得上次聊到一半没说完的那件事。系统也需要这样的积累能力。

HomeAI 的记忆分两层：本地层只属于你家，记住你们所有人的偏好、习惯、说过的每句话，隐私不出门；云端层从更多家庭的经验里提炼更好的服务方法，反哺回来——你家用得越久越懂你们，新加入的家庭起点也越来越高。

「懂你」不是把对话存下来就够了。你问 Lucas「小姨上周说的那个创业想法」——他需要知道那是小姨（不是其他人），是上周（不是去年），关于创业（不是别的话题），而且背后是她最近遇到了什么契机在推动她（所以才有那个想法）。少了任何一个细节，Lucas 都只能模糊地说「小姨好像说过什么」，说不出具体。完整记忆需要完整拼图，不只是一条对话记录。

刚装上系统那天，每一个问题都要「问外面的大脑」。用着用着，系统开始认识你家——哪些问题它自己就能答，哪些还需要往外发。时间越长，能自己答的越多，依赖外面的越少。**「越来越自给自足」的比例，就是系统专精程度的刻度盘。**

与此同时，系统在悄悄积累两类经验。一类是「懂人」的经验：哪种回应让家人满意、哪种让人失望，慢慢成为进化的养料。另一类是「做事」的经验：反复处理过的事情（收到视频链接怎么处理、账单统计怎么跑），系统会把成熟的模式结晶为可复用的技能——高频精选的技能常驻在 Agent 视野里随时可调，海量的经验沉淀在后台按需召回。好用的能力留下，笨拙的被替代，能力在使用中自然进化。

用久了，Lucas 会注意到：有些家庭成员和他说的话越来越多、越来越深。这时系统会与那个人已有的个人 AI 建立协同——对方跑在他自己的设备上，只服务他、只记他的事；Lucas 与它互联，整个家庭从一个中心节点慢慢长成一张网。越亲密的关系资源越多，长期不联系的会慢慢休眠让出空间——系统是自我生长的，但不会无边界膨胀。

两类积累慢慢融为一体。Lucas 越来越懂你们的人，也越来越可靠地做事——不需要提醒，他自己知道该做什么、怎么做才稳。

## 系统在进化，人也需要进化

系统会自己变强。经验在提炼，能力在固化，判断在优化——这些都是自动的。但有一件事不是自动的：**操盘这套系统的人，也得跟着变强**。

AI 每一代都比上一代更强。今天你还在帮系统搭架构，明天它可能已经不需要你搭了。不跟上，不是「用不好工具」的问题，而是**逐渐失去对系统方向的判断力**。跟上的方式不是看论文、学框架——而是在真实任务中与当下最强的模型持续协作，在一次次决策中训练「与 AI 共同思考」的肌肉记忆。HomeAI 的建设过程本身就是这个训练场：每一次架构决策、每一次踩坑复盘、每一次与模型的意见分歧，都在锻炼一种新的职业能力。

系统工程师的角色随系统成熟度悄悄迁移。早期你是建造者——搭架构、调管道、写配置，手感像传统工程师。系统稳定后你变成观察者——监控运行、识别缺口、偶尔干预，更像一个教练。到了最终阶段，你变成守门人——系统推来一条训练样本：「Lucas 昨天对妈妈说了这句话，这个回应好不好？」你看一眼，说「确认」或「跳过」。这不是降级。每次你确认一条样本，都在给模型权重注入你对这个家庭的判断——**什么回应是好的、什么承诺是不该做的、什么语气是这个家庭的温度**。守门人守的不是代码，是整个 AI 组织的演化方向。

这个过程不会消失。每次有价值的协作做法，AI 编码助理会主动沉淀进工程手册；每次架构决策、踩坑经验、系统干预，追加进演化档案；发现实现陷阱和平台约束，写进禁区地图让 Lisa 写代码前就知道「什么坑不能踩」。这些文档不需要人手动维护——越用越厚，第二个工程师接手就是接手一份经过验证的经验积累。

系统的进化是自动的，人的进化需要主动选择。选择持续与系统共同进化的人，最终会站在一个新的位置上：不再是写代码的人，也不再是管系统的人，而是**定义 AI 组织演化方向的人**。

## 不只是家庭 AI

如果 HomeAI 只能用在家庭，它就只是一个聪明的家庭助手。但我们要回答的问题比这大得多——三个正在困扰整个行业的真实问题。

**问题一：开发即交付，交付即开发——该是什么样的？**

做过项目交付的人都知道：每个客户的需求都是定制的。一个网络改造项目，从需求分析到概要设计到详细设计到实施方案再到验收——每一层都要专业工程师投入大量时间，每一层的产出就是交给客户的交付物。**设计过程本身就是交付过程，这两件事从来就不是分开的。**

痛苦在于：这个过程高度依赖人。方案架构师做概要设计，网络工程师做详细设计，实施工程师写割接方案——每个环节都是专业人力在燃烧。客户多了，人不够用；需求变了，前面的设计要返工；经验留在个人脑子里，下一个项目从头来过。整个行业用了二十年去优化人效——标准化模板、知识库、流程引擎——但本质没变：**核心产能被锁在人身上**。

AI 改变的不是某个环节的效率，而是产能的载体。当 AI 能理解需求、做出设计、生成方案——「开发」就不再是人的专属动作，它成了系统的内在能力。**需求浮现的那一刻系统就开始工作，系统工作的过程就是交付的过程，交付落地的那一刻反馈就进入下一次进化。** 写代码不是目的，做需求文档不是目的——利用 AI 的能力实现交付力，才是目的。

前面你已经看到了这条流水线的样子：家人说「要是有个错题本就好了」，Lucas 接住需求，Andy 出方案，Lisa 写代码，Lucas 交回来——中间没有排期、没有人工翻译环节。这条流水线验证的不是「写代码更快了」，而是「交付力可以被系统内化」。

**问题二：Agentic Organization——AI 组织应该是什么样的？**

现在很多人在谈 AI 角色、数字人、多智能体——但你仔细看，骨子里还是把 AI 当工具。给它一个人形、一个名字、一套工作流，本质上还是「人发指令，AI 执行」。工具再聪明，也只是工具。

但如果我们把这个问题推到底：**一个人类和 AI 共生的社会，到底应该怎么协作？**

不是 AI 替代人，不是 AI 辅助人，而是——AI 真正在场。在场意味着什么？意味着它理解这个组织正在发生什么，共情组织的目标，和人一起守住组织的底线和价值观，随着组织成长而成长。就像《流浪地球》里的 MOSS，但方向相反：**不是凌驾于人类之上的超级智能，而是扎根在具体组织里、和具体的人一起走的伙伴。**

当 AI 真正在场之后，变化的不是「系统能做什么」，而是**关系**——系统和人的关系，人与人之间的关系。

爸爸私聊说「姐姐最近压力大」，妈妈私聊说「黟黟好像有点焦虑」——同一件事，两个人在独立说，彼此不知道对方也在关心。一个真正在场的系统，能感知到这件事，在隐私边界内推动家人之间更好地沟通。这不是「信息聚合功能」，这是只有理解这个家庭、共情家庭目标的存在才做得到的事。

小姨在考虑创业，妈妈有相关资源但小姨不知道——系统在两人各自的对话中感知到关联，在征得双方同意后把信息连起来。这不是搜索引擎在做的事，这是一个**在场的组织成员**才会做的事。

**这才是系统进化真正在做的事。** 不是技术指标在提升，是系统越来越深地理解每个成员，每个成员越来越信任系统，两者相互适应、共同进化——系统和人的关系在变好，人与人之间的关系也因此变好。再往深走一步：系统对组织的理解从外部规则变成内化的判断力，就像一个家人不需要翻阅家规就知道什么该做什么不该做——**价值观不再是写在文件里的约束，而是长在系统里的本能**。

Andy 和 Lisa 是冲着「开发即交付」去的——他们是工具角色，让系统拥有交付力。但 Lucas 代表的是整个系统面向组织成员的那一面：在场、共情、和人一起守住底线。家人和 Lucas 说话，不是在「使用一个工具」，是在和一个理解自己、关心这个家的伙伴交流。这种理解不能被预设、不能被配置，只能从真实的共处中生长出来。

**不是替代人类，是共生涌现的携手发展。** 这是 Agentic Organization 最深处的命题：当 AI 真正共情了组织的认知和目标，系统和人、人与人之间的协作关系会自然地变得更加协调——不是被设计出来的协调，是从共处中涌现出来的协调。

**问题三：传统的研发人员，在 AI 时代何去何从？**

这个问题比大多数人理解的要大。不只是程序员在焦虑——方案架构师、网络工程师、实施工程师、项目经理，整个交付体系里的专业角色都在面对同一个现实：**你花十年积累的专业判断力，AI 正在以越来越低的成本复制。**

网络方案设计需要理解客户需求、做技术选型、出详细方案、写割接计划——这些是高度专业化的工作，是行业的核心产能。但当 AI 能做需求分析、能做方案比选、能生成配置脚本——这些专业能力就不再只能靠人来承载了。这不是「将来会发生」，是「正在发生」。

焦虑的本质是：**如果专业执行力不再是稀缺能力，工程师的价值在哪？**

一个方向正在浮现：从**自己做设计的人**，变成**设计让 AI 做出好设计的系统的人**。能力的核心从「更熟悉某个技术领域」迁移到「更懂得如何定义方向、识别系统缺口、为系统的进化方向把舵」。这不是理论推演——前面「系统在进化，人也需要进化」那一节讲的就是这个转型的实时记录。**从建造者到守门人**——守的是整个 AI 组织的演化方向。这个转型不轻松，但它是真实的、可操作的、正在发生的。比焦虑有用。

---

**这三个问题没有标准答案。** 家庭是最小的组织单元——四五个人，关系最紧密，信任最高，信息流最自然。在这么小的组织里，如果 AI 都不能从「工具」变成「成员」，在更大的组织里就更不可能。HomeAI 是在这个最小单元里做的第一次实验。

**CrewHiveClaw** 就是从 HomeAI 里提炼出来的组织 AI 化框架。三个核心机制——多角色分工、双层自进化（本地专精 + 云端赋能）、Channel 信息接入——构成了可复制的骨架。第二个组织拿到这个骨架，替换掉「家人」变成「员工」、「家庭群」变成「企业 IM」、「错题本」变成「业务报表」，同一条管道就在新场景里跑起来了。管道是通用的，流过管道的内容是每个组织自己的。

### 验证目标

**验证目标 1：本地进化能力**
> 系统能从组织信息流中积累上下文，自主交付符合组织需要的能力。用得越久，越懂这个组织——这证明管道能让 AI 在一个组织里持续生长。

**验证目标 2：可复制性**
> 另一个组织，拿到 CrewClaw 代码 + 本 Readme + AI 编码助理（CC / OpenCode 等），能按步骤独立搭建出功能相同的系统。不需要原作者介入，文档本身就够——这证明管道是组织无关的。

HomeAI 是第一个漂流瓶——完整记录设计意图、踩过的坑、做过的取舍，让第二组、第三组工程师拿到它，能在新的场景里继续回答这三个问题。每个场景都会给出不同的答案，而每个答案都在让问题本身变得更清晰。

## 如何使用本文档构建这样的系统

### 从零重建的路线图（第二个工程师必读）

```
Step 1：理解架构（1-2 小时阅读）
  本文档（Readme）→ 00-project-overview.md Part 1~3 → 04-project-constitution.md（宪法约束）

Step 2：搭建环境
  05-environment-setup.md → 按步骤安装 OpenClaw / ChromaDB / Kuzu / PM2 等依赖

Step 3：部署框架实例
  00-project-overview.md Part 3 § 十三「新实例部署 SOP」（4 步最小路径）
  → 配置 config/ 文件 + 设置环境变量 + 编写 context-sources.ts
  → 初始化 Kuzu schema（init-kuzu-schema.py）

Step 4：验证基础功能
  06-basic-version.md → 端到端测试：Lucas 收消息 → Andy 出方案 → Lisa 交付 → ChromaDB 有记录

Step 5：理解系统现状
  CLAUDE.md「下次起点」字段（当前阶段 + 待执行任务）
  09-evolution-version.md（架构决策全记录）
```

### 如果你是感性脑（业主/系统工程师）

- 读每个章节的「为什么」，理解我们在做什么
- 遇到操作步骤，按路线图把对应子文档交给 Claude Code 执行
- 负责验收：每个阶段执行完，按标准确认结果
- 完成后你就理解了这套体系，可以自主决策如何演进

### 如果你是理性脑（AI 执行者 / Claude Code）

系统工程师是人与 AI 共同构成的协作有机体——感性脑负责判断方向，理性脑负责分析与执行。

**初次接入（全新安装）**：按路线图 Step 1~4 顺序执行，Step 3 完成后按 `06-basic-version.md` 填写实例配置，Step 4 验证通过后从本 Readme 初始化 CLAUDE.md 稳定区。

**持续迭代（CLAUDE.md 已存在）**：每次新会话读「下次起点」字段定位任务；执行完成后更新 CLAUDE.md 动态区；遇到文档描述与实际环境不符，先处理差异再记录偏差。

### 文档即产品

文档是判断力过户的载体。给 Lucas 写细致的背景文档，就是把「这个组织遇到这类事，该怎么判断」写进了系统。文档写得多精确，AI 冷启动就能跑多快、起点就多高。这不是文档建设的副产品——文档**本身就是产品**，是让 AI 冷启动的最核心输入。

**本 Readme 就是这个模式最好的自我验证。** 一份完整描述了「为什么做、怎么运作、踩过什么坑、什么算成功」的文档，喂给 Lucas，他就能在这个领域冷启动。这个模式不只适用于系统搭建——网络交付、营销支撑、任何一个业务领域，只要有人写出完整上下文的任务文档，Lucas 就能把那个领域的判断力接过来，逐步从「带着我做」走向「我自己跑」。文档即产品，意味着每一份写好的任务文档，都是一个可交付、可复用、可迭代的资产。

**Readme 是人与系统之间最关键的契约**，不是配置文件，是价值观的载体——它定义了系统该是什么样、不该做什么、为什么这样设计。正因如此，Readme 的修改权不轻易下放：系统足够成熟、能诚实记录「改了什么、为什么改」时，才可以逐步让渡。这是 L4 自我进化的终极场景：AI 自主维护这份契约，每次演化留下诚实记录，同时保持对第二个复制体的完整可移植性。

每组系统工程师构建自己的实例时，往往需要先根据自己的意图刷新 Readme——先刷文档，再执行，才能更好地落实自己的意图。第二组系统工程师参照本路线图 + 第一组验证过的 Readme，可以独立搭建功能相同的系统，不需要原作者介入，文档本身就够。更多被验证过的 Readme 汇聚在一起，就是新工程师更优的地基——这是 CrewHiveClaw 架构的网络效应。
