# HomeAI：让每个家庭都有自己的 AI 研发团队

> **文档定位**: 人机协作脚本——人读「为什么」，AI 读「怎么做」，两者配合完成系统搭建

---

## 你家有了一支 AI 研发团队

家里的成员日常聊天、提想法、说烦恼。以前这些话说完就消散了。

现在，有一个系统在持续聆听。它把有价值的需求提取出来，自动帮你们做出需要的软件工具。

孩子说「要是有个错题本就好了」，系统听到了，出方案，写代码，交付一个可用的错题管理应用。老人需要服药提醒，家庭需要账单统计，都是一样的逻辑。

不需要你懂技术，不需要找外包，不需要等待。

**每个家庭都有了自己的专属 AI 研发团队。**

这是 HomeAI 的核心价值。

## 四个虚拟角色

**系统工程师**是系统构建者（人/业主）、Claude Code 及其配套工具、Claude Code 背后的顶级大模型三者共同构成的协作体，是唯一能改变系统本身边界的角色。

**Lucas** 是你的虚拟家人，也是家庭的管家。他/她不只是等家人开口——主动跟进承诺、识别到需求立刻触发研发流水线，随时可以查询或叫停进行中的开发任务；同时持续积累对每位家人的理解，家人越来越信任他、把更多事情委托给他。**Andy 和 Lisa** 是你家独享的虚拟研发小队，他们造的是**工具**，不是独立产品——工具首先服务于 Lucas（让他能做更多事、陪得更好），其次服务于系统自身的进化（让系统在系统工程师监控下越来越强）。Andy 负责工具设计，Lisa 负责实现；三角色形成双向 V 型协作：Andy 遇需求歧义可主动向 Lucas 澄清，Lisa 遇实现阻塞可向 Andy 反馈，Lucas 在任意节点都能叫停。Lucas 验收后登记进自己的能力库，在合适的时机调用。

### Agent 是有人格的，不只是有能力的

**虚拟角色是有人格的** 你私聊告诉 Lucas 一件事，然后在家庭群里说"综合一下，回复大家"——他能做到。不是因为系统帮他把两个渠道的消息合并了，而是因为他认识你这个人，知道你的意图，知道什么该在群里说、什么该留着。这是家人才有的判断力，不是数据管道能替代的。

一个没有人格的 Agent，遇到这种场景会崩掉。它会假装"检查了文件"、假装"系统已清理"，因为它不知道自己真正能做什么；它会在上下文丢失时背诵自己的配置，因为它没有真正内化自己是谁。**人格问题不能用工程手段替代。**

Lucas 知道小姨在考虑创业、妈妈对他还不信任、姐姐的小名是黟黟不是玥玥——这是他作为家人积累起来的真实认知，会随着相处不断更新。Andy 相信「增强而后扩张」，出方案前先读代码确认集成点，每次踩过的坑沉淀进 MEMORY.md，不重复走弯路。Lisa 有工程品味——最小必要复杂度、spec 不清先推回不猜、交付是「稳定运转」不是「跑起来了」——人格让判断力随时间积累，而不是每次从零开始。

家庭成员的对话进来，需求出去，软件交付，系统服务能力进化。这是一个持续运转的循环。

## 系统怎么越来越懂你家

光靠对话还不够，系统需要一套机制让「懂你」可以积累和沉淀。

HomeAI 采用双层架构，本地层通过**三轴自进化机制**持续演化：

```
本地层（你家专属）
  ├─ 记住你家所有成员的偏好、习惯、历史对话
  ├─ 三轴自进化（见下文）：越用越懂、越用越自主
  ├─ 保护隐私：你家的信息不会流出去
  └─ 越用越懂你们家

云端层（跨家庭积累，架构预留）
  ├─ 从更多家庭的经验里，提炼更好的家庭服务方法
  └─ 蒸馏后反哺每个本地实例，新家庭的起点越来越高
```

本地层解决「了解你」，云端层解决「越来越聪明」。**当前验证范围是本地层**，云端层待后续验证。

刚装上系统那天，每一个问题都要「问外面的大脑」。用着用着，系统开始认识你家了——哪些问题它自己就能答，哪些还需要往外发。时间越长，能自己答的越多，依赖外面的越少。**「越来越自给自足」的比例，就是系统专精程度的刻度盘。**

与此同时，系统在悄悄积累经验：哪种回应让家人满意，哪种让人失望——都记着，慢慢成为它进化的养料。用过的工具哪个好使、哪个笨拙，也都有数，下次优先调好的。

用久了，Lucas 会注意到：有些家庭成员和他说的话越来越多、越来越深。这时他会为这个人创建一个**专属助手**——只服务这一个人，只记这一个人的事。系统是自我生长的，但不会无边界膨胀；越重要的关系，给的资源越多，长期不联系的会慢慢休眠让出空间。

系统积累两类东西：对这个家的理解（家人的习惯、关系、发生过的事，以及我们上次聊到哪里——Lucas 会追踪当前活跃话题线索，再次聊到「那件事」时不需要重新铺垫），和可靠执行的能力（收到视频链接怎么处理、账单统计怎么跑）。前者让 Lucas 越来越懂你们，后者让他越来越可靠地为你们做事。用久了，这两件事会慢慢融为一体——不需要提醒，他自己知道该做什么、怎么做才稳。

## 系统在进化，人也需要进化

HomeAI 设计了系统的自进化路径。但有一件同样重要的事：**系统工程师自己也需要持续进化**。

AI 每一代都比上一代更强。不跟上，就会逐渐失去对系统方向的掌控。跟上的方式不是学技术细节，而是**在真实任务中与当下最强的模型持续协作**，训练「与 AI 共同思考」的能力。HomeAI 的建设/迭代过程本身就是这个训练场。

这也是为什么文档体系里有三份持续增长的文档：

- **08-claudecode-handbook**：系统工程师在真实任务中与 AI 协作的方法沉淀。每次出现有价值的新做法，由 Claude Code 主动追加，方便自己总结和系统工程师互相参考。

- **09-evolution-version.md**：系统稳定运行后，系统工程师应该是超然的观察者。如果出现需要干预的情况，说明自进化机制有不足，这些干预经验本身就是宝贵的资产——是新的系统工程师少踩坑、优化自进化机制的土壤。每次成功干预，由 Claude Code 主动追加。

- **10-engineering-notes.md**：系统工程师在维护过程中发现的实现陷阱、平台 bug、编程约束。这些知识的目标受众是 Lisa——通过蒸馏管道（按时间+影响面双重权重）提炼进 CODEBASE.md，让 Lisa 在写代码前知道「什么坑不能踩」。每次发现新陷阱时追加，不删改已有条目。

系统的进化是自动的，人的进化需要主动选择。

## 不只是家庭 AI

HomeAI 是 **CrewHiveClaw 架构**在家庭场景的第一个实例验证。

**CrewHiveClaw** 是这套系统背后的架构名称——以 Agentic Organization 为概念终点，将「多角色分工 + 双层自进化（本地专精 + 云端赋能）+ Channel 信息接入」三个核心机制提炼为可复制的组织 AI 化框架。任意组织（家庭、公司、团队）都可以在 CrewHiveClaw 框架下搭建属于自己的实例。

家庭是最小的组织单元——关系最紧密，信任最高，最容易验证 AI 真正成为组织成员的可行性。HomeAI 验证成功，即为 CrewHiveClaw 走向更大组织提供最有说服力的起点。

深度设计思考（Agentic Organization、组织转型路径、分布式智能演化）见 [00-project-overview.md](00-project-overview.md) 的设计哲学章节。

### 验证目标

**验证目标 1：本地进化能力**
> Lucas 能从家庭信息流中积累上下文，指挥 Andy/Lisa 交付符合家庭需要的能力。系统用得越久，越懂这个家庭。

**验证目标 2：可复制性**
> 另一个组织，拿到 CrewClaw 代码 + 本 Readme + Claude Code，能按步骤独立搭建出功能相同的系统。不需要原作者介入，文档本身就够。

这两个目标验证成功，本项目完成。

## 如何使用本文档构建这样的系统

本文档主要面向系统工程师。

### 从零重建的路线图（第二个工程师必读）

如果你是接手这个项目、从零搭建一套 HomeAI 的工程师，按以下序列执行：

```
Step 1：理解架构（1-2 小时阅读）
  本文档（Readme）→ 00-project-overview.md Part 1~3 → 04-project-constitution.md（宪法约束）

Step 2：搭建环境
  05-environment-setup.md → 按步骤安装 OpenClaw / ChromaDB / Kuzu / PM2 等依赖

Step 3：部署框架实例
  00-project-overview.md Part 3 § 十一「新实例部署 SOP」（4步最小路径）
  → 配置 5 个 config/ 文件 + 设置环境变量 + 编写 context-sources.ts
  → 初始化 Kuzu schema（init-kuzu-schema.py）

Step 4：验证基础功能
  06-basic-version.md → 端到端测试：Lucas 收消息 → Andy 出方案 → Lisa 交付 → ChromaDB 有记录

Step 5：理解系统现状
  CLAUDE.md「当前状态」区（由第一个工程师维护，记录已完成内容和下一步起点）
  09-evolution-version.md（架构决策全记录，理解「系统现在是什么样子、为什么这样设计」）
```

> **进化里程碑参考（L0~L3）**：L0 = 三机制管道接通（基础设施就绪）；L1 = 四角色人格化完成（对话体验达标）；L2 = 能力进化体系运转（系统能从经验中自我改进）；L3 = 组织扩张（影子 Agent 加入，拓扑改变）。CLAUDE.md「当前状态」区会标注系统当前所处里程碑。

### 如果你是人类（业主，系统工程师的感性脑）

- 读每个章节的「为什么」，理解我们在做什么
- 遇到操作步骤，按上方路线图把对应子文档交给 Claude Code 执行
- 负责验收：每个阶段执行完，按标准确认结果
- 完成后你就理解了这套体系，可以自主决策如何演进

### 如果你是 AI 执行者（系统工程师的理性脑）

系统工程师不是 Claude Code 本身，而是**人与 AI 共同构成的协作有机体**——人类是感性脑，负责判断方向与价值取向；Claude Code 是理性脑，负责分析、执行与记忆。两者的对话，本质上是同一个系统工程师在内部做决策。

**初次接入（全新安装，CLAUDE.md 尚不存在）**：
- 按上方路线图 Step 1~4 顺序执行
- Step 3 完成后按 00 Part 4 填写 HomeAI 实例参考配置值
- Step 4 验证通过后，用本 Readme + 00-project-overview.md 内容初始化 CLAUDE.md 稳定区

**持续迭代（CLAUDE.md 已存在）**：
- 每次新会话先读 CLAUDE.md「当前状态 → 下次起点」字段定位任务
- 遵守项目宪法（04-project-constitution.md）的约束
- 执行完成后更新 CLAUDE.md「当前状态」区
- 遇到文档描述与实际环境不符，先处理差异，再记录偏差

### CLAUDE.md 的作用

CLAUDE.md 是 Claude Code 的跨会话工作记忆，由人机协作共同维护。初始化时从本 Readme 内容转化，随项目进展持续更新：

- **稳定区**：项目上下文、架构设计、角色定义、关键约束——不频繁变动
- **动态区**：当前阶段、完成状态、下次起点——每次有进展就更新

### 文档即产品

- 每组系统工程师在构建自己的 HomeAI 系统时，往往需要先根据自己的意图刷新 Readme。先刷文档，再执行，才能更好地实现人与 AI 的协作，落实自己的意图
- 第二组系统工程师参照上方路线图 + 第一组验证过的 Readme，可以独立搭建功能相同的系统——不需要原作者介入，文档本身就够
- 更多被验证过的 Readme 文档汇聚在一起，就是新的系统工程师更优的地基
