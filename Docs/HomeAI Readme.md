# HomeAI：让每个家庭都有自己的 AI 研发团队

> **文档定位**: 人机协作脚本——感性脑读「为什么」，理性脑读「怎么做」，两者配合完成系统搭建与持续进化

---

## 你家有了一支 AI 研发团队

家里的成员日常聊天、提想法、说烦恼。以前这些话说完就消散了。

现在，有一个系统在持续聆听。它把有价值的需求提取出来，自动帮你们做出需要的软件工具。

孩子说「要是有个错题本就好了」，系统听到了，出方案，写代码，交付一个可用的错题管理应用。老人需要服药提醒，家庭需要账单统计，都是一样的逻辑。不需要你懂技术，不需要找外包，不需要等待。

用久了，这个系统开始认识你家每一个人——说话的习惯、关注的事、没说完的话。家人越来越信任它，委托给它的事越来越多，它也越来越懂这个家庭。有一天，它会为每个深度参与的家庭成员配置一个专属 AI 助理，跑在那个成员自己的设备上——整个家庭形成一张 AI 协作网络，而不只是一个中心点。

**每个家庭都有了自己的专属 AI 研发团队。这是 HomeAI 的核心价值。**

## 三个问题，一个实验

这个系统不只是家庭 AI 工具。它试图回答三个正在困扰整个行业的真实问题。

**问题一：开发即交付，交付即开发——该是什么样的？**

做过项目交付的人都知道：每个客户的需求都是定制的。一个网络改造项目，从需求分析到概要设计到详细设计到实施方案再到验收——每一层都要专业工程师投入大量时间，每一层的产出就是交给客户的交付物。**设计过程本身就是交付过程，这两件事从来就不是分开的。**

痛苦在于：这个过程高度依赖人。方案架构师做概要设计，网络工程师做详细设计，实施工程师写割接方案——每个环节都是专业人力在燃烧。客户多了，人不够用；需求变了，前面的设计要返工；经验留在个人脑子里，下一个项目从头来过。整个行业用了二十年去优化人效——标准化模板、知识库、流程引擎——但本质没变：**核心产能被锁在人身上**。

AI 改变的不是某个环节的效率，而是产能的载体。当 AI 能理解需求、做出设计、生成方案——「开发」就不再是人的专属动作，它成了系统的内在能力。**需求浮现的那一刻系统就开始工作，系统工作的过程就是交付的过程，交付落地的那一刻反馈就进入下一次进化。** 写代码不是目的，做需求文档不是目的——利用 AI 的能力实现交付力，才是目的。

HomeAI 的 Lucas → Andy → Lisa 流水线，是这个问题的第一次真实回答。家人说「要是有个错题本就好了」，Andy 出方案、Lisa 写代码、Lucas 交回来——中间没有排期、没有人工翻译环节。这条流水线验证的不是「写代码更快了」，而是「交付力可以被系统内化」。

**问题二：Agentic Organization——AI 组织应该是什么样的？**

现在行业里谈 AI，大多还是「AI 辅助」——Copilot 帮补代码，ChatGPT 帮写文档，本质上还是人干活、AI 打下手。工具更好用了，但组织运转方式没变：还是那些人、那些角色、那些协调成本。

但你有没有想过一个更根本的问题：如果 AI 不只是工具，而是团队里真正的角色呢？不是「辅助设计师出方案的助手」，而是**自己能判断该不该做这个设计、怎么做、做完了主动找人审**的角色。如果组织里多了这样的成员——从不休息、记得所有上下文、能同时处理多条线——整个组织应该怎么运转？汇报关系怎么变？协调机制怎么变？人的位置在哪里？

没有人真正回答过这个问题，因为没有人真正试过。现有的 multi-agent 框架大多是工作流编排——本质上还是脚本，只是节点换成了 LLM。节点之间没有真正的协作关系，没有冲突解决，没有信任积累，没有组织记忆。

HomeAI 试的是另一条路：Lucas、Andy、Lisa 不是流水线上的三个节点，是有分工、有边界、有冲突解决机制的组织成员。Lucas 浸入信息流持续理解组织节奏，Andy 主动发现系统缺口不等人催，Lisa 有工程判断力遇到不清楚的 spec 会推回而不是硬猜。三者之间的协调方式，随系统成熟从「直接监督」走向「工作流标准化」再走向「相互调整」，最终到「价值观内化」——**L0 到 L4 就是这条组织成熟路径在系统层的映射**。

**问题三：传统的研发人员，在 AI 时代何去何从？**

这个问题比大多数人理解的要大。不只是程序员在焦虑——方案架构师、网络工程师、实施工程师、项目经理，整个交付体系里的专业角色都在面对同一个现实：**你花十年积累的专业判断力，AI 正在以越来越低的成本复制。**

网络方案设计需要理解客户需求、做技术选型、出详细方案、写割接计划——这些是高度专业化的工作，是行业的核心产能。但当 AI 能做需求分析、能做方案比选、能生成配置脚本——这些专业能力就不再只能靠人来承载了。这不是「将来会发生」，是「正在发生」。

焦虑的本质是：**如果专业执行力不再是稀缺能力，工程师的价值在哪？**

一个方向正在浮现：从**自己做设计的人**，变成**设计让 AI 做出好设计的系统的人**。能力的核心从「更熟悉某个技术领域」迁移到「更懂得如何定义方向、识别系统缺口、为系统的进化方向把舵」。这不是理论推演——HomeAI 的建设过程就是这个转型的实时记录：系统工程师的日常工作，随着系统从 L0 走向 L4，从搭架构、调管道，变成监控运行、识别缺口，最终变成审核 DPO 样本、决定什么判断值得进入模型权重。**从建造者到守门人**——守的是整个 AI 组织的演化方向。

这个转型不轻松，但它是真实的、可操作的、正在发生的。比焦虑有用。

---

**这三个问题没有标准答案。** HomeAI 是在真实家庭场景里做的第一次实验——完整记录设计意图、踩过的坑、做过的取舍，让第二组、第三组工程师拿到这个「漂流瓶」，能在新的场景里继续回答。每个场景都会给出不同的答案，而每个答案都在让问题本身变得更清晰。

**系统工程师**是人（业主感性脑）与 AI 工具（Claude Code 理性脑）共同构成的协作体，是唯一能改变系统本身边界的角色。

## 协作设计：从 Vibe 到 Harness

### V 字协作链

需求从 Lucas 进来，经过 Andy 的设计，到 Lisa 的实现，最后交回 Lucas 验收——这是一个 V 字形。V 字型的好处是：需求和交付都在 Lucas，Andy/Lisa 是黑盒；Lucas 可以在任意节点叫停，Andy 可以推回澄清需求，Lisa 可以推回阻塞问题，整条链路双向流动。

Andy 和 Lisa 造的是**工具**，不是独立产品——工具首先服务于 Lucas（让他能陪伴更好、做更多事），其次服务于系统自身的进化。

Lucas 还会为深度参与的家庭成员配置专属 AI 节点——真正跑在那个成员自己设备上的助理，只服务他、只记他的事。节点通过家庭知识图谱与 HomeAI 中心协作，让整个家庭从「一个 AI」变成「AI 网络」。**节点属于成员本人，不属于 Andy/Lisa，管理权只属于 Lucas。** 这是角色边界，不只是实现细节——节点是私人助理，不是系统的执行工具，归属错了成员就失去了控制权。

### 入口接得住 Vibe，出口交得出 Harness

好的系统不是等用户把需求想清楚了再来。大多数时候，家人就是一句"我想做点什么"，甚至只是觉得哪里不对劲。大模型最擅长的不是执行清晰指令——那是脚本干的事——它擅长的是接住那个模糊的 vibe，帮人一步步理清楚，最后做出一个连提需求的人自己都没想到能做成这样的事。

"harness"是系统给用户的标准——严谨、可控、可验证。"vibe"是用户真实的状态——模糊、试探、甚至自己都不确定要什么。**入口接得住 vibe，出口交得出 harness** ——这是 Lucas + Andy + Lisa 三角色协作链的设计目标。Lucas 负责接住 vibe（陪伴澄清、理清意图），Andy + Lisa 负责交出 harness（结构化设计 + 可验证交付）。

### Agent 是有人格的，不只是有能力的

你私聊告诉 Lucas 一件事，然后在家庭群里说"综合一下，回复大家"——他能做到。不是因为系统帮他把两个渠道的消息合并了，而是因为他认识你这个人，知道你的意图，知道什么该在群里说、什么该留着。这是家人才有的判断力，不是数据管道能替代的。

一个没有人格的 Agent，遇到这种场景会崩掉。它会假装"检查了文件"、假装"系统已清理"，因为它不知道自己真正能做什么；它会在上下文丢失时背诵自己的配置，因为它没有真正内化自己是谁。**人格问题不能用工程手段替代。**

Lucas 知道小姨在考虑创业、妈妈对他还不信任、姐姐的小名是黟黟不是玥玥——这是他作为家人积累起来的真实认知，会随着相处不断更新。Andy 相信「增强而后扩张」，出方案前先读代码确认集成点，每次踩过的坑沉淀进 MEMORY.md，不重复走弯路；他不只是被动出方案，还主动从系统运行信号中发现问题，从「任务执行者」进化为「系统思考者」。Lisa 有工程品味——直接编码优先，最小必要复杂度，spec 不清先推回不猜，交付是「稳定运转」不是「跑起来了」——这是判断力，不是工具列表。

对话进来，需求浮现，设计成型，代码交付，能力沉淀——然后下一轮对话进来，系统已经比上一轮更懂你了。这不是一次性的软件交付，是一个持续加速的循环。

## 系统怎么越来越懂你家

光靠对话还不够。人记住一个家庭成员，靠的是日积月累的相处——记得她说话的方式，记得他最近在忙什么，记得上次聊到一半没说完的那件事。系统也需要这样的积累能力。

HomeAI 的记忆分两层：本地层只属于你家，记住你们所有人的偏好、习惯、说过的每句话，隐私不出门；云端层从更多家庭的经验里提炼更好的服务方法，蒸馏后反哺回来——你家用得越久越懂你们，新加入的家庭起点也越来越高。**当前验证范围是本地层**，云端层待后续验证。

「懂你」不是把对话存下来就够了。你问 Lucas「小姨上周说的那个创业想法」，系统要知道这句话关于什么**话题**（语义），是**什么时候**说的（时间），涉及**谁**（实体），**为什么**她有这个想法（因果）。四个维度缺一个，Lucas 都只能泛泛地想起「小姨好像说过什么」——精准回忆依赖完整的维度覆盖。技术细节见 00-project-overview.md 第六节。

刚装上系统那天，每一个问题都要「问外面的大脑」。用着用着，系统开始认识你家——哪些问题它自己就能答，哪些还需要往外发。时间越长，能自己答的越多，依赖外面的越少。**「越来越自给自足」的比例，就是系统专精程度的刻度盘。**

与此同时，系统在悄悄积累两类经验。一类是「懂人」的经验：哪种回应让家人满意、哪种让人失望，慢慢成为进化的养料。另一类是「做事」的经验：反复处理过的事情（收到视频链接怎么处理、账单统计怎么跑），系统会把成熟的模式结晶为可复用的技能——高频精选的技能常驻在 Agent 视野里随时可调，海量的经验沉淀在后台按需召回。好用的能力留下，笨拙的被替代，能力在使用中自然进化。

用久了，Lucas 会注意到：有些家庭成员和他说的话越来越多、越来越深。这时他会为这个人配置一个**专属 AI 节点**——真正跑在那个成员个人设备上的助理，只服务他、只记他的事，随时可以被他直接调用。这不是一个数字标签，而是一个活的实体；注册到家庭知识图谱后，Lucas 在协调任务时就知道「这件事可以调那台设备处理」，整个家庭 AI 网络从一个中心节点变成了一张网。节点属于成员本人——这是私人助理，不是系统的执行工具。系统是自我生长的，但不会无边界膨胀；越重要的关系给的资源越多，长期不联系的会慢慢休眠让出空间。

两类积累慢慢融为一体。Lucas 越来越懂你们的人，也越来越可靠地做事——不需要提醒，他自己知道该做什么、怎么做才稳。

## 系统在进化，人也需要进化

系统会自己变强。蒸馏管道在跑，Skill 在结晶，路由阈值在优化——这些都是自动的。但有一件事不是自动的：**操盘这套系统的人，也得跟着变强**。

AI 每一代都比上一代更强。今天你还在帮系统搭架构，明天它可能已经不需要你搭了。不跟上，不是「用不好工具」的问题，而是**逐渐失去对系统方向的判断力**。跟上的方式不是看论文、学框架——而是在真实任务中与当下最强的模型持续协作，在一次次决策中训练「与 AI 共同思考」的肌肉记忆。HomeAI 的建设过程本身就是这个训练场：每一次架构决策、每一次踩坑复盘、每一次与模型的意见分歧，都在锻炼一种新的职业能力。

系统工程师的角色随系统成熟度悄悄迁移。早期你是建造者——搭架构、调管道、写配置，手感像传统工程师。系统稳定后你变成观察者——监控运行、识别缺口、偶尔干预，更像一个教练。到了 L4，你变成守门人——Main 推来一条 DPO 样本，你看一眼，说「确认」或「跳过」。这不是降级。每次你确认一条样本，都在给模型权重注入你对这个家庭的判断——**什么回应是好的、什么承诺是不该做的、什么语气是这个家庭的温度**。本地微调闭环已经接通，这个转型正在真实发生。守门人守的不是代码，是整个 AI 组织的演化方向。

这个过程会留下痕迹。文档体系里有三份持续增长的文档，记录的就是这条成长曲线：

- **08-claudecode-handbook**：系统工程师与 AI 协作的方法沉淀。每次出现有价值的新做法，由 Claude Code 主动追加——这是「人怎么用好 AI」的实战手册。

- **09-evolution-version.md**：每次架构变更或干预经验的全记录。理解「每个机制为什么存在」的核心档案；也是新工程师少踩坑的历史地图——这是「系统怎么走到今天」的来时路。第二个工程师重建时，Step 1 阅读阶段过一遍这份文档，知道「前人踩过哪些坑」比「自己踩了才知道」省太多时间。

- **10-engineering-notes.md**：发现的实现陷阱、平台 bug、不可绕过约束（组件析构崩溃、API 静默截断、行为规则不注入等于不存在……）。目标受众是 Lisa——通过蒸馏管道提炼进 CODEBASE.md，让她在写代码前知道「什么坑不能踩」——这是「系统的禁区地图」。

系统的进化是自动的，人的进化需要主动选择。选择持续与系统共同进化的人，最终会站在一个新的位置上：不再是写代码的人，也不再是管系统的人，而是**定义 AI 组织演化方向的人**。

## 不只是家庭 AI

如果 HomeAI 只能用在家庭，它就只是一个聪明的家庭助手。但我们要回答的问题比这大得多：**AI 能不能真正成为一个组织的成员？**

家庭是最小的组织单元——四五个人，关系最紧密，信任最高，信息流最自然。在这么小的组织里，如果 AI 都不能从「工具」变成「成员」，在更大的组织里就更不可能。反过来，如果在家庭里验证成功了，同一套机制就有充分理由走进企业、团队、学校——因为底层逻辑是相同的：浸入信息流、理解组织成员、按需生长能力、在服务中进化。

**CrewHiveClaw** 就是从 HomeAI 里提炼出来的这套组织 AI 化框架。三个核心机制——多角色分工、双层自进化（本地专精 + 云端赋能）、Channel 信息接入——构成了可复制的骨架。第二个组织拿到这个骨架，替换掉「家人」变成「员工」、「家庭群」变成「企业 IM」、「错题本」变成「业务报表」，同一条管道就在新场景里跑起来了。管道是通用的，流过管道的内容是每个组织自己的。

深度设计思考（Agentic Organization、组织转型路径、分布式智能演化）见 [00-project-overview.md](00-project-overview.md) 设计哲学章节。

### 验证目标

**验证目标 1：本地进化能力**
> Lucas 能从家庭信息流中积累上下文，指挥 Andy/Lisa 交付符合家庭需要的能力。系统用得越久，越懂这个家庭——这证明管道能让 AI 在一个组织里持续生长。

**验证目标 2：可复制性**
> 另一个组织，拿到 CrewClaw 代码 + 本 Readme + Claude Code，能按步骤独立搭建出功能相同的系统。不需要原作者介入，文档本身就够——这证明管道是组织无关的。

这两个目标验证成功，本项目完成。HomeAI 是第一个漂流瓶，里面装的不只是一个家庭 AI 的故事，而是一整套经过真实验证的组织 AI 化方法论。

### 从零重建的路线图（第二个工程师必读）

```

Step 4：验证基础功能
  06-basic-version.md → 端到端测试：Lucas 收消息 → Andy 出方案 → Lisa 交付 → ChromaDB 有记录

Step 5：理解系统现状
  CLAUDE.md「下次起点」字段（当前阶段 + 待执行任务）
  09-evolution-version.md（架构决策全记录）
```

### 如果你是感性脑（业主/系统工程师）

- 读每个章节的「为什么」，理解我们在做什么
- 遇到操作步骤，按路线图把对应子文档交给 Claude Code 执行
- 负责验收：每个阶段执行完，按标准确认结果
- 完成后你就理解了这套体系，可以自主决策如何演进

### 如果你是理性脑（AI 执行者 / Claude Code）

系统工程师是人与 AI 共同构成的协作有机体——感性脑负责判断方向，理性脑负责分析与执行。

**初次接入（全新安装）**：按路线图 Step 1~4 顺序执行，Step 3 完成后按 Part 4 填写实例参考配置，Step 4 验证通过后从本 Readme 初始化 CLAUDE.md 稳定区。

**持续迭代（CLAUDE.md 已存在）**：每次新会话读「下次起点」字段定位任务；执行完成后更新 CLAUDE.md 动态区；遇到文档描述与实际环境不符，先处理差异再记录偏差。

### 文档即产品

每组系统工程师构建自己的实例时，往往需要先根据自己的意图刷新 Readme——先刷文档，再执行，才能更好地落实自己的意图。

第二组系统工程师参照本路线图 + 第一组验证过的 Readme，可以独立搭建功能相同的系统，不需要原作者介入，文档本身就够。更多被验证过的 Readme 汇聚在一起，就是新工程师更优的地基——这是 CrewHiveClaw 架构的网络效应。

> **进化里程碑（L0~L4）**：技术细节与实现规范见 00-project-overview.md，这里只记每层最重要的「为什么」和「坑在哪里」，供第二个系统工程师理解设计来时路。
>
> **L0 = 基础设施 Ready**：管道先于质量——四维记忆结构（语义/时间/实体/因果）的写入通道必须在做质量之前定义清楚。不是跑通功能，是「信息能被正确捕获」。坑：ChromaDB 超 500 条静默丢失；对话不分块向量语义稀释；Kuzu 析构 SIGBUS。
>
> **L1 = Agent 行为质量 + 自进化**：「好不好」比「能不能」更难。Skill 固化不能靠模型自觉——弱信号预标记 → Agent 主动结晶 → 基础设施自动结晶，三层都需要，单靠其中一层都不够。坑：行为规则不注入等于不存在；承诺幻觉无法靠模型自省解决。
>
> **L2 = Engineering Anything**：本质是「稳定输出 + 过程可见 + 可选干预」，不是「把 CLI 体验搬到即时通信」。关键设计取舍：Lisa 必须有直接编码工具（工具层确定性高于 LLM 迭代）；状态同步必须走 Lucas 不能绕过成员理解层；审批用超时默认（即时通信渠道阻塞等待会卡死任务）。坑：jiti 缓存让代码修改静默失效。
>
> **L3 = 组织协作进化**：带着人一起进化。核心设计决策：成员私有 AI 节点属于成员本人，不属于 Andy/Lisa——节点是私人助理，不是系统工具；归属错了成员就失去控制权，违背「带着人」原则。坑：节点注册不需要独立服务，Gateway 内存注册表足够，独立服务只增加故障点。外围成员（访客/合作伙伴）享受完整 L2 体验，但不可引导 L3/L4 演化方向。
>
> **L4 = 系统自进化**：Andy 是 L4 的乘数——spec 精度每提升一点，整条 L2 链成功率都上升；L4 投入应集中在 Andy。「正朔→认知」是 L4 的控制回路——系统工程师修改 Readme 影响 Agent 认知，Agent 行为产生新数据，L4 提炼新能力，新能力反馈给系统工程师。本地微调闭环（LoRA→fuse→GGUF→Ollama）已接通，自动调度运行。L2 帮人做事，L4 自己变强；L3 带着人，L2 不带——这三个差别是设计决策，不能混淆。
