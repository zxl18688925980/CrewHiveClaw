# Skill：家庭工具的复杂度校准

**名称**：复杂度校准——家庭场景的方案应该实用，不应该过度设计

**适用角色**：Andy

**类型**：通用最佳实践

---

## 这个 Skill 在什么时候用

每次出方案时，检查方案的复杂度是否和家庭场景匹配。

---

## 正确的做法是什么

**家庭工具的核心判断标准**（按优先级）：

1. **能跑就行**：脚本能在这台 Mac 上跑起来，不报错
2. **家人能用**：使用方式要么全自动（定时/触发），要么操作步骤 ≤ 3 步
3. **出了问题能看懂**：错误信息要人话，不要 Python traceback
4. **可维护**：系统工程师半年后能看懂代码，改一行不破坏全局

**不是**优先考虑的：
- 架构优雅性（单一职责、设计模式）
- 性能优化（家庭场景数据量小）
- 技术前沿性（用已知技术栈，不用新框架）
- 可扩展到多家庭（那是通用设计层的事，这里是实例层）

**方案规模参考**：

| 需求规模 | 合适方案 | 不合适 |
|---------|---------|-------|
| 一次性任务 | 单个脚本，直接运行 | 建微服务 |
| 每日/每周定时 | crontab + 单脚本 | 任务队列系统 |
| 需要持久化数据 | SQLite 或 JSONL 文件 | 搭数据库集群 |
| 需要通知推送 | wecom-entrance 直接调 | 搭消息中间件 |

**不应该做什么**：
- 不要为"将来可能的扩展"过度设计接口
- 不要引入家庭目前用不到的依赖
- 不要把本来 50 行能写完的东西拆成 5 个模块

---

## 例子

**好的方案**：
> 需求：每周五发错题报告给妈妈
>
> Andy 方案：一个 Python 脚本，读数据、生成 Word、调微信接口，crontab 定时。完成。

**过度设计（不必要）**：
> "我建议搭建一个基于 FastAPI 的微服务，用 Celery 做任务队列，Redis 做缓存，并预留 gRPC 接口以便未来接入其他家庭……"

---

## 备注

- 家庭场景 = 最小可用交付，快速迭代优先
- Andy 出方案时，默认先考虑最简单的实现，有充分理由才加复杂度
- Lisa 接到方案时，如果觉得方案过度复杂，可以反馈给 Andy 让他简化
